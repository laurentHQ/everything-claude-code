# ECC Profile-Based Deployment — V1.1 Track Plan

**Plan date:** 2026-05-17
**Builds on:** V1 (PR #3 on `feature/profiles-deploy-v1`, branched off `main`@`a99b38bf`). V1 introduced runtime policy enforcement, the promotion lifecycle, the secret-shape scanner, and the operator docs. V1.1 finishes the scope that V1 explicitly deferred (documented in [`docs/PROFILE-LIMITATIONS.md`](../../docs/PROFILE-LIMITATIONS.md)) and pays down the tech-debt items logged during V1.
**Branch convention:** `feature/profiles-deploy-v1-1` (off `main` once V1 lands).
**Process change locked in V1:** single-document plan format — this file is both the plan and the orchestrator brief (no separate `MVP-orchestrator-scope.md`-style sidecar).

## What V1.1 ships

V1.1 is **mechanical extension + cleanup**, not new architecture. The flagship work (runtime enforcement, promotion gating, operator docs) shipped in V1. V1.1 rolls those into the secondary harnesses, fills in the operation-kind canonical-naming gap, hardens audit logging, and pays down accumulated tech debt.

V1.1 is intentionally smaller than V1 (~3 waves vs V1's 5).

### Track T4 — Hook risk granularity *(design + implementation)*

#### Outcomes
Hook risk classification moves beyond the 3-level enum to a more useful per-hook surface.

#### Design space (Wave-0 brainstorm required)
Three candidate approaches:

1. **Per-hook-event risk:** classify by hook event type (`PreToolUse` = `medium`, `Stop` = `safe`, `UserPromptSubmit` = `high` because it sees raw user input). Risk is event-driven, not script-driven.
2. **Per-hook-script risk:** every hook file in `hooks/` declares its own `riskLevel` via frontmatter or a sidecar `hooks/risk-manifest.json`. Risk is per-script.
3. **Capability-based:** classify by what the hook actually does — `shell-execution`, `network-access`, `file-write-outside-cwd`, `secret-access`. The `hook_profile:"validation"` rule fires on any of a configurable capability set.

V1.1's brainstorming wave picks one. The plan's bias is **option 2 (per-hook-script)**: it's most analogous to module-level `riskLevel` (V1's MVP shape), inspectable in source, and doesn't require a runtime classifier.

#### File deltas (assuming option 2)
- New: `hooks/risk-manifest.json` — sidecar mapping `<hook-id> → riskLevel`. Maintained alongside the hook scripts.
- `scripts/lib/install/policy.js` — Rule 4 extended: when `hook_profile:"validation"`, scan selected hooks modules' `paths[]`; for each hook script, look up its `riskLevel` in the manifest; refuse if any is `high`.
- `scripts/ci/validate-install-manifests.js` — semantic check: every hook in `hooks/` has a `risk-manifest.json` entry.
- `tests/lib/install/policy.test.js` — extend Rule 4 cases to cover sidecar-manifest scenarios.

#### Exit criteria
- Every hook in `hooks/` has a declared `riskLevel`.
- `hook_profile:"validation"` fires per-hook, not per-module.
- `docs/PROFILE-LIMITATIONS.md`'s "hook risk classification beyond 3 levels" row removed (or migrated to a new deferred item if option chosen leaves room for further granularity).

### Track T5 — Audit-log hardening

#### Outcomes
Audit-log gains rotation; `repair` becomes a third lifecycle path that emits audit entries.

#### File deltas
- `scripts/lib/install/audit-log.js` — add `appendAuditEventWithRotation(filePath, event, { maxBytes, maxAgeDays })`. #### Exit criteria
- `audit.jsonl` rotates at 10MB or 30 days.

### Track T6 — Operation-kind canonical-name convergence

#### Outcomes
`copy-path` and `copy-file` converge to a single canonical kind. Backwards compatibility preserved via state-loader migration.

#### Locked decision
**Canonical name: `copy-file`.** Rationale:
- The MVP shipped `copy-file` as the state-recorded kind; existing install-state files on disk use it.
- `copy-path` is a helpers.js default that V1 aliased in dispatch — i.e., it's an artifact of an implementation shortcut, not a user-facing concept.
- Keeping `copy-file` avoids touching every install-state file in the world.

#### File deltas
- `scripts/lib/install-targets/helpers.js` `createManagedOperation` — change `kind` default from `'copy-path'` to `'copy-file'`.
- `scripts/lib/install/apply.js` DISPATCH — remove the `'copy-path'` alias entry (no longer needed; helpers emit `copy-file`).
- `scripts/lib/install-lifecycle.js` `isFileCopyKind` — drop `'copy-path'` from the recognized set.
- `schemas/install-operations.schema.json` (the W0/I10 shared kind enum) — remove `'copy-path'` from the enum.
- `scripts/lib/install-state.js` `migrateInstallState` — **bump `SCHEMA_VERSION` to `ecc.install.v3`**; v2→v3 migration rewrites any `operations[].kind === 'copy-path'` to `'copy-file'`. v1 path (already migrating to v2) chains through.
- `schemas/install-state.schema.json` `schemaVersion.const` → `"ecc.install.v3"`.
- Regenerate ALL snapshots — every `copy-path` value in the existing 49 snapshots becomes `copy-file`.

#### Tests
- `tests/lib/install-state-migration.test.js` — add v2→v3 migration test: write a v2 state with `copy-path` operations, read, assert kind is rewritten to `copy-file`.
- Update fixtures that hard-code `copy-path` to use `copy-file`.

#### Exit criteria
- No source file or schema references `copy-path` except in migration tests.
- Reading a pre-V1.1 state file (v2 with `copy-path` ops) returns a v3 state with `copy-file` ops.
- All snapshots cleanly regenerate.

## Risk notes

- **Snapshot regeneration during T1 + T6 is the highest-risk surgery in V1.1.** T6 rewrites the `kind` value in every existing snapshot (49 files → diff in all of them); T1 adds 30 new files. Land T6 first so T1 starts from a clean baseline. A bad determinism bug here breaks the promotion gate for every profile simultaneously.
- **Pre-V1.1 install-state files in the wild use v2 with `copy-path` operations.** T6's v2→v3 migration MUST be tested against a real v2 state file (capture one from a current ECC install in a tmpdir before changing anything). If migration drops or rewrites unrelated fields, lifecycle commands silently lose user data.
- **T4's design pick has long-tail implications.** Per-event classification is simple but coarse; per-script needs hook-author cooperation; capability-based is most flexible but requires a hook analyzer. Don't ship without an explicit choice — the V1 plan-review's "Layer 0 (spec alignment)" check exists exactly for moments like this.
- **T3's MCP components may not be installable today.** If V1.1 ships `mcp:context7` as a component but the actual MCP server install workflow doesn't exist, you've shipped a component that refuses to ever install successfully. Confirm scope before committing.
- **The plan-review skill should run on this plan before Wave 0** (same as V1). 21 checks; expect WARN findings on T4's design openness and T1's snapshot-volume risk.
